## 5つの結論

1. **CodexはチャットAIではなく「仕事の置き場所」になった** — OpenAI公式PDFが"仕事の文脈をどこへ置くか"という新しい問いを提示。Durable Threadsがその実装形態だ。
2. **Thread automationsで「準備して待つ」スタイルが主流になる** — 30分ごとのSlack監視・週次サマリー生成など、ハートビート型の自律実行がCodexの本来の使い方として公式化された。
3. **「ループを設計する人が勝つ」という認識がエコシステムで広まった** — Peter Steinberger（220万ビュー）・Boris Cherny・@taiyo_ai_gakuseが独立して同じ結論に達した。ループエンジニアリングが次の競技場だ。
4. **人間の役割は「承認と不可逆な判断だけ」に集約されている** — 探す・読む・整理する・下書きするはCodexの仕事。人間が担うのは送信・公開・責任を持つという不可逆アクションのみ、と公式PDFが明示している。
5. **50%超の利用がコーディング以外になった** — 400万超の週間アクティブユーザーのうち、半分以上がコードを書かない用途でCodexを使っている。製品アイデンティティが「コーディングツール」から「知的仕事の代行者」に移行した。

## 第1章 「仕事の置き場所」というパラダイムシフト

OpenAI Codex担当者のJason Liuが著した公式PDF「Codex-maxxing for long-running work」（2026年公開、27ページ・10章）は、Codexの使い方についての根本的な問い直しから始まる。

✅（出典：OpenAI公式PDF）
- 「最初に考えるべきは"どんなプロンプトを書くか"ではなく**"この仕事の文脈をどこへ置くか"**」
- 単発プロンプトでCodexを使う場合、毎回同じコンテキストを渡し直す必要がある
- 長く続く仕事には「専用スレッド」が必要だ、という考え方をCodex-maxxingと定義している

🔶（出典：@kawai_design）
- Codexをチャットとして使うとき、同じプロジェクト情報を何度も伝え直す摩擦が生まれる
- 解決策は**コンテキストをスレッドに「置く」こと**——会話の連続性ではなく、仕事の継続性を単位にする発想の転換

---

### Durable Threads——コンテキストを失わずに再開する

10機能の中で最初に紹介されるのがDurable Threads（永続スレッド）だ。これは単なる「会話履歴の保存」ではない。

✅（出典：OpenAI公式PDF）
- **仕事専用の継続スレッド**として機能する。「新しいチャット」を開くたびにコンテキストが消えない
- プロジェクトごと・顧客ごと・ワークストリームごとに専用スレッドを作ることが推奨される
- スレッドはそのまま**Thread automationsの実行基盤**にもなる

🔶（出典：@kawai_design）
- 長期スレッド向きのタスク例：毎週続くプロジェクト・顧客対応・OSSのissue確認・リリースノート作成
- 「チャットを閉じるたびにプロジェクトが死ぬ問題」を根本的に解決する機能

---

### 最初に作る3ファイル

Durable Threadsを開くだけでは不十分で、スレッドに「置く情報」を設計することが重要だ。

🔶（出典：@kawai_design）
- **`workstream.md`** — 現在のタスク・優先順位・進行状況のサマリー
- **`memory/project-state.md`** — プロジェクトの背景情報・決定済み事項・避けるべきこと
- **`done-definition.md`** — 完了条件の定義。「いつ終わったとみなすか」を事前に書く

この3ファイルをスレッド開始時にCodexに渡すことで、再開のたびにゼロから説明する必要がなくなる。

## 第2章 10機能の実装詳解

公式PDFは「Durable threads → Voice → Steering → Memory → Computer use → Remote control → Thread automations → Three loops → Goals → Side panel」という10章構成で、機能単位に使い方を解説している。

---

### Voice input——スピードではなく「質の変化」

✅（出典：OpenAI公式PDF）
- 音声入力の価値は「タイピングより速い」ではなく、**整理されていない思考をそのまま渡せる**ことにある
- 頭の中で文章化するプロセスを省略できるため、指示の質と速度が同時に上がる
- 「考えながら話す」情報の伝達が可能になり、ブレインダンプ的な使い方に適している

---

### Steering——作業中に方向修正する

✅（出典：OpenAI公式PDF）
- Codexが作業中に**次の指示・修正・方向転換を追加**できる機能
- 作業を止めずに「こっちより先にあっちをやって」「この方針は変更する」と割り込める
- 長時間タスクで方向がズレたときに最初からやり直さずに済む

---

### Memory / vault——会話の外に記憶を置く

✅（出典：OpenAI公式PDF）
- **メモリは会話ウィンドウの外に置く**——会話が終わっても記憶が消えない設計
- `vault/` というディレクトリ構造が推奨されており、以下のようなファイルを置く：
  - 個人の好みや作業スタイル
  - プロジェクト固有の知識・用語集
  - 過去の決定と理由
- Durable Threadsと組み合わせることで「仕事の文脈が完全に引き継がれる」状態を実現できる

---

### Computer & Browser use——3種の使い分け

✅（出典：OpenAI公式PDF）
Codexが使えるブラウザ・コンピュータアクセスには3種類あり、場面によって使い分ける。

| 識別子 | 用途 | 特徴 |
|--------|------|------|
| `$browser` | ローカルブラウザ | 認証不要のパブリックサイト向け |
| `@chrome` | 認証済みChromeセッション | ログイン済みサービスへのアクセス |
| `@computer` | GUIアプリ全般 | デスクトップアプリの操作 |

✅（出典：OpenAI公式PDF）
- 認証が必要なサービス（Gmail・Slack・社内ツール等）には`@chrome`を使う
- この3種の使い分けを理解しないと、ログインが必要なサイトへのアクセスが失敗する

---

### Remote control——別デバイスから承認する

✅（出典：OpenAI公式PDF）
- PCから開始した長時間タスクをスマートフォンから**確認・承認・方向修正**できる機能
- Codexが「人間の承認が必要な判断」に達したときに通知を送り、人間が別デバイスで応答できる
- 「朝起きたらスマホでセッション確認」「外出中にCodexを管理」というワークフローが可能に

🔶（出典：@kawai_design）
- Remote controlはThread automations・Steeringと組み合わせることで真価を発揮する
- 「Codexが準備する → 人間がモバイルで承認する → Codexが実行する」という非同期ループが実現する

---

### Thread automations——ハートビート型の自律実行

Thread automationsはこの公式発表の中でも実用的な応用範囲が最も広い機能だ。

✅（出典：OpenAI公式PDF）
- スレッドに**定期実行のスケジュール**を設定できる機能
- 代表例：30分ごとにSlack・Gmailをチェックして重要情報をまとめる
- 「人間がPCの前にいなくても仕事が進む」状態を作る
- Three loopsの実装基盤になる

🔶（出典：@kawai_design）
- 週次サマリー・競合情報の自動収集・カスタマーサポートへの初回返答草案など、反復タスクがすべて候補
- ハートビート型（定期実行）とイベント型（何かが起きたら）の2パターンが想定されている

---

### Side panel——スレッド内でレビューする

✅（出典：OpenAI公式PDF）
- Markdown・CSV・スプレッドシート・スライドをCodexスレッド内でレビューできる機能
- 外部ツールを切り替えずに「生成 → 確認 → 修正指示 → 再生成」のサイクルが完結する
- 長文ドキュメントや表形式データを扱うタスクで特に有効

## 第3章 3つのループ例とGoals設計

公式PDFのChapter 8「Three loops」は、Thread automationsを使った実践的なループ設計の具体例を3つ示している。これが「長時間タスクをどう設計するか」の教科書になっている。

---

### Chief of Staff ループ

✅（出典：OpenAI公式PDF）
- Codexが「首席補佐官」として機能するループ
- 毎朝：受信トレイ確認 → タスクの優先順位付け → 一日の計画を提示する
- 毎週：プロジェクト状況の集約 → 週次レポート生成 → 翌週の準備
- 人間は優先順位の確認と最終承認だけを担当する

---

### フィードバック監視ループ

✅（出典：OpenAI公式PDF）
- 製品・サービスへのフィードバックを常時監視するループ
- Slack・Twitter・サポートチャネル・レビューサイト等を30分ごとにチェック
- 重要なフィードバックを検出 → 分類 → 担当者への通知 → 週次サマリー生成
- 「Codexが常にアンテナを張っている」状態を作る

---

### 返金対応ループ

✅（出典：OpenAI公式PDF）
- カスタマーサポートの一形態として返金リクエストを処理するループ
- 受信 → 情報整理 → 対応草案生成 → **人間に送信前確認を求める**
- 「送信する」という不可逆アクションの直前で必ず人間が介在する設計になっている
- これがCodexの役割分担原則（後述）の実装例

---

### 弱いゴール vs 強いゴール

Goals（Chapter 9）はループ設計において最も重要な章だ。ゴールの書き方で成果が大きく変わる。

✅（出典：OpenAI公式PDF）
**弱いゴールの例**：「毎週レポートを作って」
- 何のレポートか、誰向けか、何を含めるか、いつ終わりと判断するかが不明

**強いゴールの4要素**：
1. **完了条件** — 「〇〇の状態になったら終わり」を明示
2. **制約** — 「〜はしない」「〜の範囲内でやる」という境界線
3. **検証基準** — 成果物が正しいかどうかをどう判定するか
4. **エスカレーション条件** — 人間に判断を求めるのはどのタイミングか

強いゴールを書くことで、Codexが自律的に正しい方向に進み続けられる。

## 第4章 Loop Engineeringという概念の出現

公式発表と並行して、「ループを設計する」という概念がエコシステムで独立して盛り上がりを見せている。OpenAIの公式発表の裏側でユーザーコミュニティが同じ結論に達していたことが、このタイミングを特別なものにしている。

---

### Peter Steinberger「プロンプトではなくループを設計しろ」

⚠️（出典：@taiyo_ai_gakuse の分析）
- Peter Steinbergerの投稿「エージェントにプロンプトを打つな。ループを設計しろ」が220万ビューを記録
- ルールとして書くのではなく、**繰り返しを前提とした仕組みを作る**ことを主張
- この投稿と@AnatoliKopadze（182万ビュー、3エージェント40分ビルド）が同時期に広まった

🔶（出典：@AnatoliKopadze）
- Anthropicのエンジニアが3エージェント（計画担当・構築担当・判定担当）でゼロからフルアプリを40分で構築するデモを公開
- 「勝者は最も賢いモデルを持っている人ではなく、**最良のループを持っている人**」という発言が拡散

---

### ループ成立の4条件

⚠️（出典：@taiyo_ai_gakuse の分析）

ループが実際に機能するために必要な4条件が整理されている。

| 条件 | 内容 |
|------|------|
| ① タスクが繰り返す | 1回限りではなく、週次・日次・常時で発生し続けるタスクであること |
| ② 検証が自動化できる | 成果物が正しいかどうかをコードまたはCodex自身が判定できること |
| ③ トークン予算に余裕がある | 長時間実行に必要なコンテキスト量をまかなえるコスト設計であること |
| ④ エージェントが適切な道具を持つ | ループに必要なツール（ブラウザ・API・ファイル操作等）がすべてアクセス可能であること |

---

### ralphループ → /goal → オーケストレーションという系譜

⚠️（出典：@taiyo_ai_gakuse の分析、注意：新規性については後述の未確認事項を参照）

「ループエンジニアリング」は突然現れた概念ではなく、進化の系譜がある。

- **2025年：ralphループ** — コミュニティ発のループパターン。エージェントが自律実行を繰り返す構造
- **2026年：/goalコマンド** — Codexの公式機能としてループを設定する仕組みが整備される
- **現在：オーケストレーションループ** — 複数エージェントが分担・連携して動く構造（計画・実行・判定の分離）

⚠️ 注意：@taiyo_ai_gakuse自身も「新規性は少し盛られている」と指摘している。Anthropicが2024年12月に文書化済みの**evaluator-optimizerパターン**の延長線上にある概念であり、完全に新しいアイデアではない。

## 第5章 非コーディング利用の拡大と役割分担の新標準

---

### 400万ユーザーと50%超の非コーディング利用

🔶（出典：Codex公式ブログ、2026年5月時点）
- **週間アクティブユーザー：400万人超**
- **コーディング以外のユースケースが50%超**に到達

50%超という数字は、Codexの製品ポジションが「コーディングツール」から「知的仕事全般の代行者」に変化したことを示している。実際に公式PDFが対象とする用途は、コードを一切含まない業務（顧客対応・レポート作成・情報収集・フィードバック監視など）だ。

🔶（出典：@kawai_design）
- 長期スレッド向きの実例として挙げられているタスクの多くはコーディング以外：
  - 週次の顧客対応スレッド
  - 競合情報の定期収集
  - OSSのissueトリアージ
  - リリースノートの自動生成

---

### Codexがやること / 人間がやること

公式PDFの最も重要なフレームワークが、この役割分担の明示化だ。

✅（出典：OpenAI公式PDF）

**Codexがやること（自律実行）**：
- 探す・読む・整理する・下書きする・選択肢を作る

**人間がやること（不可逆な判断）**：
- 判断する・承認する・**送信する**・**公開する**・責任を持つ

太字の「送信する」「公開する」がここでのポイントだ。返金対応ループ・フィードバック監視ループ・Chief of Staffループのいずれも、「最後のアクション」は人間が行う設計になっている。Codexは「準備が完了したので確認してください」とエスカレーションし、人間がGoボタンを押す。

🔶（出典：@kawai_design）
- この役割分担は`done-definition.md`に明示的に書いておくことが推奨されている
- 「Codexが完了を宣言する前に何をすべきか」「どの判断だけを自分がする」を事前定義しておくと自律実行の精度が上がる
